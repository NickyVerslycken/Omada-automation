Dependency and relation map (files, classes, functions)
=======================================================

Legend
- `A -> B`: A depends on / calls / uses B.
- `uses`: direct import or invocation.

2) Class/function ownership and usage direction
-----------------------------------------------

`omada_batch/ui/shell/app_shell.py`
- contains class `App` (extends `customtkinter.CTk`).
- orchestrates sidebar navigation, header/status bar, queue polling (`_poll_queue`), threaded operations (`_run_bg`), and log output (`_log`).
- handles developer mode toggle via sidebar switch and page show/hide via `_show_page`.
- formats log/dev-json output with numbered lines and truncation policy.
- composes behavior by inheriting:
  - `ConnectionControllerMixin`
  - `NetworksControllerMixin`
  - `BatchControllerMixin`
  - `VlanBatchControllerMixin`

`omada_batch/ui/state/app_state.py`
- contains class `AppState`.
- owned by `App` as `self.app_state` (renamed from `self.state` to avoid conflict with CTk's `.state()` method).
- stores shared mutable fields (`client`, `sites`, `gateways`, `plan`, caches, profiles, etc.).

`omada_batch/ui/controllers/connection_controller.py`
- contains class `ConnectionControllerMixin`.
- handles profile CRUD/import/export, controller auth/session connect, site list and disconnect flow.
- forwards API request/response JSON payloads into queue event type `devjson`.
- writes profile credentials into `data/.env` via config helper using profile-ID-scoped env keys and stores only env-key references in profile JSON.
- removes unreferenced credential env keys from `data/.env` when a profile is deleted.
- uses changelog-aware JSON writer when exporting controller profiles without raw secrets.

`omada_batch/ui/controllers/networks_controller.py`
- contains class `NetworksControllerMixin`.
- handles gateway/device refresh, network fetch/filter/export, and gateway-to-network matching logic.
- filters DHCP server choices to gateway-class devices before populating comboboxes.
- gateway/network matcher now checks additional ownership keys such as `gatewayMac` and `assignIpDeviceMac`.
- triggers inline batch-tab interface selector refresh when gateway selection/state changes.
- uses changelog-aware JSON writer when exporting LAN networks.

`omada_batch/ui/controllers/batch_controller.py`
- contains class `BatchControllerMixin`.
- handles plan generation, preview export, push loop, and push-button enablement state.
- manages inline per-network interface/port selection table in Batch Create > Networks sub-page (replacing popup selection).
- consumes WAN/selectable metadata from interface catalog and blocks WAN port selection in UI + payload mapping.
- uses changelog-aware JSON writer when exporting batch plans.

`omada_batch/ui/controllers/vlan_batch_controller.py`
- contains class `VlanBatchControllerMixin`.
- handles VLAN-only batch plan generation, gateway port selection, and push loop via `/networks/confirm` endpoint.
- manages inline per-VLAN port selection table in Batch Create > VLANs sub-page.
- uses gateway device MAC and port IDs for VLAN network creation without DHCP.

`omada_batch/storage/file_change_log.py`
- contains changelog helpers for create/edit/delete filesystem events.
- centralizes JSONL logging to `backup/file_changelog.jsonl` by default.
- includes per-entry rollback instructions to undo logged changes.
- includes git-worktree synchronization helper to log file changes outside direct app write paths.
- normalizes recorded paths to project-relative format (`Omada-automation/...`) with `/`.
- resolves concrete timestamped backup file references for rollback metadata when available.

`omada_batch/ui/tabs/*.py`
- contains pure view-builder functions.
- each `build_*_tab(app)` creates widgets and wires callbacks to controller methods.
- `batch_tab.py` hosts a segmented button switcher for batch sub-pages (Networks, VLANs, extensible for future types).
- `batch_networks_tab.py` contains the Networks sub-page (LAN creation with DHCP).
- `batch_vlan_tab.py` contains the VLANs sub-page (VLAN creation without DHCP).

`omada_batch/api/omada_client.py`
- contains class `OmadaOpenApiClient`, used by connection and batch workflows.
- emits optional request/response JSON debug payloads.
- uses preview truncation helper so short lines are not suffixed with `...`.
- uses version fallback preference `v3 -> v2 -> v1` for key endpoint families where available.
- enriches site device results using `networks/devices/ports` endpoint fallback so interface selection can use natural port names.

`omada_batch/services/*.py`
- contain pure-ish domain helpers invoked from controllers.

`omada_batch/config.py`
- provides `data/.env` parsing/update/delete utilities used by profile normalization/save/remove workflows.

3) Core data-flow direction
---------------------------

Connect flow
1. Connection tab input -> `ConnectionControllerMixin.on_connect`.
2. Creates `OmadaOpenApiClient`, authenticates, loads sites.
3. Pushes queue events -> `App._poll_queue` -> updates widgets/state.

Gateway/network flow
1. Site select -> refresh gateways.
2. Device list -> `get_device_port_details` enrichment -> `build_interface_catalog` -> gateway combos.
3. Network list -> `_apply_network_filter` based on gateway/device/interface matches.

Batch Networks flow
1. Batch parameters -> `generate_plan` -> `AppState.plan`.
2. Interface mapping configured inline in Batch Create > Networks sub-page with WAN ports visible but disabled.
3. Per-plan push -> `OmadaOpenApiClient.create_lan_network`.

Batch VLANs flow
1. VLAN parameters -> `generate_vlan_plan` -> `AppState.vlan_plan`.
2. Port mapping configured inline in Batch Create > VLANs sub-page per gateway device.
3. Per-plan push -> `OmadaOpenApiClient.create_vlan_network` via `/networks/confirm`.

4) Dependency characteristics
-----------------------------

- Orchestration center moved from one large class to a composed shell + mixins.
- Tab modules are now view-only builders.
- Business/UI workflow logic is grouped by domain in controllers.
- Shared mutable runtime state is centralized in `AppState`.
- `main_window.py` remains as compatibility shim for existing imports.
